**Advanced Architecture**

Parameter Store is a capability of Systems Manager that allows you to store parameter values as key-value pairs. It can store up to 4 MB per value, supports hierarchies using paths, and provides secure storage options.

Internally, Parameter Store uses Amazon [[AWS_SA_PRO_Obsidian_Notes/Master/S3|S3]] to store the parameters in a highly available and durable way across multiple regions. The metadata such as path, type, and versioning information are stored in [[dynamodb|DynamoDB tables]].

When dealing with large datasets or global applications, there are some aspects to keep in mind:

- Hierarchy organization: Use paths to create a hierarchy that makes it easier to organize your parameters.
- Global distribution: To achieve low latency worldwide, consider using the "Standard" parameters with [[kms]] encryption instead of the SecureString type. This way, they will be replicated across all [[ssm]] endpoints around the world.
- Data size: For data larger than 4MB, consider using [[AWS_SA_PRO_Obsidian_Notes/Master/S3|S3]] bucket references or AWS [[Storage Gateway]].

**Comparison & Anti-Patterns**

| Service          | Use Cases                                                              |
| --------------- | -------------------------------------------------------------------- |
| Parameter Store | Configuration management, storing sensitive data, small application settings |
| [[Srinivas_Notes/S3|S3]]              | Large object storage, file sharing, static website hosting            |
| [[Git_hub_notes/AWS-SAP-C02-Notes-main/README|Secrets Manager]] | Encryption key management, storing and rotating secrets             |

Anti-patterns include:

- Using Parameter Store to manage large objects or files.
- Storing database credentials directly in application code or configuration files.
- Not encrypting sensitive data when using Parameter Store.

**[[appsync|Security]] & Governance**

Complex [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|IAM]] [[policies]] may involve allowing specific users or roles to read or write parameters based on their function within the organization. Here's an example JSON policy snippet:

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "ssm:GetParameters",
                "ssm:GetParametersByPath"
            ],
            "Resource": "arn:aws:ssm:*:123456789012:parameter/*"
        },
        {
            "Effect": "Deny",
            "Action": [
                "ssm:GetParameters",
                "ssm:GetParametersByPath"
            ],
            "Resource": "arn:aws:ssm:*:123456789012:parameter/prod/*",
            "Condition": {
                "StringNotEqualsIfExists": {
                    "aws:SourceVpce": "vpce-1234abcd"
                }
            }
        }
    ]
}
```

Cross-account access can be granted by creating a role in the source account that allows the destination account's [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|IAM]] entity to assume it. Then, in the destination account, allow the necessary permissions to interact with the parameters.

Organization Service Control [[policies]] (SCPs) can enforce restrictions on actions related to Parameter Store. An example would be restricting the usage of `ssm:PutParameter*` to ensure only authorized users can update sensitive data.

**Performance & Reliability**

Throttling limits depend on the type of parameters used. Standard parameters have a limit of 50 requests per second while Advanced parameters (SecureString) have a limit of 10 requests per second. If these limits are exceeded, consider implementing exponential backoff strategies.

High availability and [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|disaster recovery]] patterns should follow [[iam|best practices]] for Systems Manager, which includes deploying resources in different Availability Zones and monitoring performance metrics through [[cloudwatch]].

**[[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|Cost Optimization]]**

Granular cost controls in Parameter Store come from understanding the pricing model and applying proper tagging. There are no direct costs for storing parameters, but charges may apply for API calls made to Parameter Store. Tagging helps identify which services are generating these costs and enables better allocation of expenses.

For example, if you have two teams A and B, you could assign tags like `{"Team": "A"}` and `{"Team": "B"}` to each team's respective set of parameters. By doing so, you can monitor and control costs generated by each team individually.

**Professional Exam Scenarios**

Scenario 1:
You are working on a multi-account environment with strict [[appsync|security]] requirements. Your company has a central [[appsync|Security]] Team managing shared resources and a Development Team working on several projects. How would you implement a solution to securely store and share sensitive configuration data between accounts?

Correct answer: Implement a centralized [[parameter-store|AWS Systems Manager Parameter Store]] in the [[appsync|Security]] Team's account. Grant necessary permissions to the Development Team members using [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|IAM]] roles and [[policies]]. Ensure that sensitive data is encrypted at rest using [[kms]] keys.

Incorrect answer: Share the sensitive data by hardcoding them into application code or using an unencrypted [[AWS_SA_PRO_Obsidian_Notes/Master/S3|S3]] bucket.

Scenario 2:
Your application needs to fetch sensitive data from [[parameter-store|AWS Systems Manager Parameter Store]], but sometimes experiences high latencies due to distance from the endpoint. What steps can you take to improve latency without compromising [[appsync|security]]?

Correct answer: Use the "Standard" parameters with [[kms]] encryption instead of the SecureString type. This way, the system will automatically replicate the data across all [[ssm]] endpoints around the world, reducing latency.

Incorrect answer: Copy the same parameters manually to multiple regions or duplicate the data in another centralized location.
