## [[RDS_Instance_Types|1. Advanced Architecture]]

GuardDuty is a regional-based threat detection service that continuously monitors for malicious activity and unauthorized behavior in your AWS accounts and deployed resources. It operates by analyzing AWS [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|CloudTrail]] event logs, [[vpc-flow-logs|VPC Flow logs]], and DNS query logs. The service uses machine learning, statistical analysis, and anomaly detection techniques to identify unusual and potentially harmful activities.

### [[RDS_Instance_Types|Global Scale Considerations]]

GuardDuty has built-in support for multi-account management through [[organizations|AWS Organizations]], enabling you to centrally manage findings and configure settings across multiple member accounts. This allows you to create a [[appsync|security]] boundary around your organization and monitor all member accounts from a single GuardDuty master account.

### Under the Hood Mechanics

Internally, GuardDuty consists of several components responsible for processing data, generating findings, and providing notifications. These include:

- *Data ingestion*: GuardDuty collects log data from various sources like [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|CloudTrail]], [[vpc-flow-logs|VPC Flow Logs]], and DNS queries. Data is encrypted using AWS Key Management Service ([[kms]]) keys during transmission.
- *Threat intelligence feeds*: GuardDuty incorporates threat intelligence feeds provided by AWS and other trusted third parties to detect known malicious IP addresses, domains, and file hashes.
- *Machine learning models*: GuardDuty employs machine learning algorithms to analyze the collected data, identify patterns, and generate risk scores for detected threats.
- *Findings*: Findings generated by GuardDuty contain information about the suspected threat, including resource ID, severity level, type, description, and timestamps. They can be exported to Amazon [[AWS_SA_PRO_Obsidian_Notes/Master/S3|S3]], [[lambda|AWS Lambda]], or Amazon [[cloudwatch]] Events for further analysis and response actions.

## [[RDS_Instance_Types|2. Comparison & Anti-Patterns]]

| Service | Use Cases |
| --- | --- |
| GuardDuty | Real-time monitoring of AWS environments, threat detection, and compliance reporting. |
| [[shield|AWS Shield]] | Protecting applications against distributed denial-of-service (DDOS) attacks. |
| AWS WAF | Filtering web traffic based on predefined rules and custom [[cloudformation|conditions]]. |
| [[network-and-dns-firewall|AWS Network Firewall]] | Stateful firewall for inspecting and filtering network traffic. |

### Anti-patterns

- Using GuardDuty as a primary defense mechanism instead of employing multiple layers of [[appsync|security]] controls.
- Monitoring only specific types of events or resources, leading to partial visibility into potential [[appsync|security]] threats.
- Configuring overly permissive [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|IAM]] [[policies]], allowing unauthorized users to modify GuardDuty findings or configurations.

## [[RDS_Instance_Types|3. Security & Governance]]

Complex [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|IAM]] [[policies]] involve granting cross-account access permissions between GuardDuty master and member accounts. Here's an example JSON policy snippet:

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Principal": {
                "AWS": "arn:aws:iam::ACCOUNT_ID:root"
            },
            "Action": [
                "guardduty:CreateMembers",
                "guardduty:DeleteMembers",
                "guardduty:ListMembers",
                "guardduty:GetMembers",
                "guardduty:UpdateMembers",
                "guardduty:GetMaster",
                "guardduty:ListDetectors"
            ],
            "Resource": "arn:aws:guardduty:*:ACCOUNT_ID:detector/*"
        }
    ]
}
```

Cross-account access can also be managed using Organization Service Control [[policies]] (SCPs):

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "guardduty:CreateMembers",
                "guardduty:DeleteMembers",
                "guardduty:ListMembers",
                "guardduty:GetMembers",
                "guardduty:UpdateMembers",
                "guardduty:GetMaster",
                "guardduty:ListDetectors"
            ],
            "Resource": [
                "*"
            ],
            "Condition": {
                "StringEquals": {
                    "aws:SourceVpc": "vpc-xxxxxxxxxxxxx"
                }
            }
        }
    ]
}
```

## [[RDS_Instance_Types|4. Performance & Reliability]]

GuardDuty imposes throttling limits on API calls and data ingestion rates. To avoid hitting these limits, implement exponential backoff strategies when making repeated requests. For high availability and [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|disaster recovery]], distribute GuardDuty resources across multiple regions and ensure adequate replication of associated infrastructure such as [[vpc-flow-logs|VPC Flow Logs]] and DNS query logs.

## [[RDS_Instance_Types|5. Cost Optimization]]

Granular cost controls for GuardDuty can be achieved by adjusting data ingestion settings, setting up [[billing]] alarms, and utilizing tagging for cost allocation. Calculation examples:

- Enabling data ingestion for specific IP addresses and limiting the number of days for which GuardDuty retains logs.
- Setting up [[billing]] alarms using [[cloudwatch|CloudWatch Alarms]] and [[billing|AWS Budgets]].
- Tagging GuardDuty resources and applying [[cost-allocation-tags|cost allocation tags]] at the [[organizations|AWS Organizations]] level.

## [[RDS_Instance_Types|6. Professional Exam Scenarios]]

Scenario 1: A large enterprise manages multiple AWS accounts within an [[AWS Organization]]. They want to enable centralized threat detection and real-time monitoring while minimizing costs. How would you design their solution?

Correct Answer: Enable GuardDuty in the master account, add member accounts to the detector, and apply relevant data ingestion settings. Implement [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|cost optimization]] strategies like adjusting data retention periods and using [[billing]] alarms.

Incorrect Answers: Implementing GuardDuty in each individual account, configuring GuardDuty to process unnecessary data sources, or ignoring [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|cost optimization]] options.

Scenario 2: A financial services firm needs to secure their AWS environment against cyber threats while ensuring regulatory compliance. What tools should they use in addition to GuardDuty?

Correct Answer: In addition to GuardDuty, use [[shield|AWS Shield]] for DDoS protection, AWS WAF for filtering web traffic, and [[network-and-dns-firewall|AWS Network Firewall]] for stateful inspection of network traffic.

Incorrect Answers: Relying solely on GuardDuty, misconfiguring [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|IAM]] [[policies]], or failing to incorporate additional [[appsync|security]] measures.
