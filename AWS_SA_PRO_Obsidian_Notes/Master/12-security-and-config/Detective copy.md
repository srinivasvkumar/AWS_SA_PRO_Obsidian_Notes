### [[RDS_Instance_Types|1. Advanced Architecture]]

[[Detective]] is a service that helps you perform cross-account [[appsync|security]] investigations, analyze and visualize event timelines and understand the scope of a [[appsync|security]] issue. It uses a powerful query language called "Event Patterns" to search for events across accounts and regions.

At its core, [[Detective]] consists of three main components:

* **Findings** represent potential [[appsync|security]] issues or risks in your environment. They are created by running queries against Event Patterns.
* **Events** are records of specific activities in your AWS environment, such as API calls made to AWS services.
* **Behavior graphs** visually represent the relationships between entities (such as [[ec2]] instances, [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|Lambda functions]], or [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|IAM]] users) involved in an event.

[[Detective]] operates at a global level, but it stores events and findings per region. This means that if you have resources spanning multiple regions, you will need to run your queries separately for each region.

### [[RDS_Instance_Types|2. Comparison & Anti-Patterns]]

| Service | Use Cases |
|---|---|
| [[Detective]] | Cross-account [[appsync|security]] investigation, visualizing relationships between entities, analyzing suspicious behavior. |
| [[Git_hub_notes/AWS-SAP-C02-Notes-main/README|CloudTrail]] | Centralized [[vpc-flow-logs|logging]], auditing, compliance reporting, and operational troubleshooting. |
| [[GuardDuty]] | Real-time threat detection based on machine learning and anomaly detection. |

Anti-pattern: Using [[Detective]] as a real-time monitoring solution. [[Detective]] excels at historical analysis rather than real-time monitoring. Services like [[GuardDuty]] are better suited for this purpose.

Cross-account access can be granted using [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|IAM]] roles and trust [[policies]] that allow specific actions for the [[Detective]] APIs. Organization SCPs can enforce centralized control over which AWS accounts within the organization can create, delete, or modify detectives.

### [[RDS_Instance_Types|4. Performance & Reliability]]

[[Detective]] has throttling limits on its API operations. For example, the `DescribeFindings` operation has a limit of 200 requests per second. To handle these [[AWS_SA_PRO_Obsidian_Notes/Master/12-security-and-config/cloudhsm|limitations]], implement exponential backoff strategies when making repeated requests.

HA/DR patterns involve deploying [[Detective]] in multiple regions to ensure high availability and redundancy. Since [[Detective]] stores events and findings per region, replicating data across regions requires additional infrastructure and automation.

### [[RDS_Instance_Types|5. Cost Optimization]]

Granular cost controls include setting up [[billing]] alerts for [[Detective]] usage, enabling [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|cost optimization]] recommendations via Trusted Advisor, and using [[organizations|AWS Organizations]] to apply [[cost-allocation-tags|cost allocation tags]].

For example, suppose you want to allocate costs based on the number of findings generated by [[Detective]] in each account. In that case, you could create a custom cost allocation report tagged with the member account ID and the type of finding (e.g., anomalous behavior, unauthorized access).

### [[RDS_Instance_Types|6. Professional Exam Scenarios]]

#### Scenario A

You are designing a secure multi-account architecture for a large enterprise. The company wants to use [[Detective]] to investigate potential [[appsync|security]] issues across all their AWS accounts. However, they don't want individual accounts to accidentally delete or modify the [[detective]] setup.

Which of the following solutions would best address this requirement?

a) Create a separate AWS account for [[Detective]] and grant permissions only to authorized personnel.
b) Implement Service Control [[policies]] (SCPs) within the [[organizations|AWS Organizations]] to restrict access to [[Detective]].
c) Set up a dedicated [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|IAM]] user with programmatic access to manage [[Detective]].

Correct answer: b) Implement Service Control [[policies]] (SCPs) within the [[organizations|AWS Organizations]] to restrict access to [[Detective]].

Explanation: By implementing SCPs, you can enforce centralized control over which AWS accounts within the organization can create, delete, or modify detectives. This ensures that individual accounts cannot accidentally delete or modify the [[detective]] setup while still allowing authorized personnel to manage it.

Incorrect answers:

a) Creating a separate AWS account for [[Detective]] may not resolve the issue since individual accounts can still be granted permissions to modify the [[detective]] setup.

c) Setting up a dedicated [[Master/Git_hub_notes/AWS-SAP-C02-Notes-main/README|IAM]] user with programmatic access to manage [[Detective]] does not provide sufficient protection against accidental deletion or modification by individual accounts.

#### Scenario B

Your organization has enabled [[Detective]] in all production accounts. Recently, you noticed that some [[Detective]] API operations exceeded their throttling limits. As a result, the affected users received [[api-gateway|errors]] and had to retry their requests.

To avoid these [[api-gateway|errors]], you decide to implement an exponential backoff strategy. Which of the following steps should you take first?

a) Increase the frequency of [[Detective]] API calls.
b) Monitor the throttling limits of [[Detective]] API operations.
c) Decrease the frequency of [[Detective]] API calls.

Correct answer: b) Monitor the throttling limits of [[Detective]] API operations.

Explanation: Before implementing an exponential backoff strategy, it's essential to monitor the throttling limits of [[Detective]] API operations. This allows you to identify which operations are being throttled and determine the appropriate backoff algorithm.

Incorrect answers:

a) Increasing the frequency of [[Detective]] API calls would worsen the problem by causing more requests to be throttled.

c) Decreasing the frequency of [[Detective]] API calls might help avoid throttling [[api-gateway|errors]], but it doesn't optimize the performance of the system. An exponential backoff strategy provides a more efficient solution by intelligently adjusting the request rate based on the observed throttling limits.
